# 統計方法學檢核表(模式B,可重用範本)

稽核方固定範本。本專案的稽核分兩種獨立模式,**可分開執行、互不取代**:模式A「數據查核」(見 `數據查核程序.md`,驗證數字是否真的算出來的)與本檔模式B「方法學確認」(驗證算法本身是否恰當)。完整稽核 = 兩者都跑;單獨跑任一模式時,紀錄中必須標明另一模式尚未執行。本表審的是方法選擇,不是計算對錯。

## 執行原則(省 token / 提高精準度)

1. **風險分級**:只對「陽性結論」(聲稱顯著/存活/成立,會誘導行動的結果)做完整檢核;否證結論僅做 Stage 1 輕掃。
2. **漏斗式執行**:Stage 0 圈定陽性結論 → Stage 1 Grep 抽決策行(不整檔 Read)→ Stage 2 單點敏感度計算(主對話直接跑 `python -c`,不派 fork)→ Stage 3 僅 🔴 候選才派 fork 深查。
3. **每項判定綁「結論會翻轉嗎?」測試**:瑕疵存在但換穩健方法後結論不變 → 🟡;陽性結論消失/翻轉 → 🔴。判定必附敏感度數字,禁止「文獻上有更好做法」式空泛評語。
4. 判定分三級:✅ 方法恰當 / 🟡 可接受但有更穩健替代(附敏感度數字) / 🔴 方法錯用且足以翻轉結論。

## 檢核項目

| # | 檢核點 | 偵測方式(Stage 1) | 判斷規則 | 敏感度計算(Stage 2,僅標黃時) |
|---|---|---|---|---|
| M1 | 多重比較家族界定(look-elsewhere) | grep 各腳本 FDR 呼叫範圍;數全專案總檢定次數 | 各子研究自劃家族 → 至少該做全專案層級敏感度 | 把所有 p 值併成一族重算 global FDR,看陽性結論是否存活 |
| M2 | 檢定相依結構 | 讀 bh_fdr 實作一次;判斷檢定間是否同股同期高度相關 | 高度相依下 BH 可能過鬆,標準替代為 Benjamini-Yekutieli | 用 BY 門檻(BH 除以 Σ1/i)重算存活數 |
| M3 | 重疊窗口自相關 | grep `lag`、持有期定義 | HAC lag ≥ 持有期;lag 是否事前定死而非事後調整 | lag 加倍重算 NW p 值,看是否仍 <0.05 |
| M4 | 分布假設 vs 資料實態 | grep `pearsonr`/`corrcoef`;實測訊號與報酬欄位峰態 | 峰態>>3(厚尾)時 Pearson 對離群值敏感,應補秩相關 | 對陽性結論重算 Spearman,方向與顯著性是否保持 |
| M5 | 訓練/測試切分的事前性 | grep split 常數;查 git log 中 split 首次出現時間 vs 測試結果首次產出時間 | 切分需來自固定規則(如最後1/3)或可證明早於結果 | 無法量化時如實標「⚠️ 推論:無法從程式碼證明事前性」 |
| M6 | 小樣本漸近檢定 | grep 結果 CSV 的 N 欄位 | 事件型 N<30 不應報漸近 p 值,只能描述性記錄 | 檢查文件是否誠實標註樣本不足 |
| M7 | 效果量揭露的對稱性 | 對照:陽性結論有無 R²/效果量?否證結論的揭露密度是否相同? | 只在方便處揭露效果量 = 選擇性呈現 | 對每個陽性結論補算 R²,與文件敘述比對 |
| M8 | 選擇性報告 | 結果 CSV 總列數 vs 文件提及的檢定數 | 文件只提顯著者而未交代總檢定數 → 🟡 | 數 CSV 全量 vs 文件敘述覆蓋率 |

## 動態對抗測試(M9~M11,測 pipeline 行為而非文字)

M1~M8 為靜態審查,抓不到「程式碼文字看起來正常但行為有漏洞」的問題(微妙的資料洩漏、merge 帶入未來資料、split 邊界差一日、樣本靜默流失)。以下三項直接測試管線行為,成本約各一輪管線重跑,同樣依風險分級選用:

| # | 檢核點 | 做法 | 判讀 | 適用對象 |
|---|---|---|---|---|
| M9 | 負對照/安慰劑 | 餵「不可能有訊號」的輸入重跑整條管線:事件日換隨機假日期、訊號序列跨日期重排(打斷時序保留分布) | 假輸入也跑出顯著 → 🔴 管線有洩漏或機制性偏誤(不需知道 bug 在哪) | 優先打陽性結論 |
| M10 | 合成資料檢定力 | 在隨機資料中注入已知強度的人工訊號,看管線能否找回 | 找不回 → 🔴 管線某處靜默毀掉訊號,「否證」結論可能是管線壞掉而非市場無訊號 | 優先打否證結論(唯一能檢驗否證的工具) |
| M11 | 擾動脆弱性 | 研究者自由度參數微擾各重跑:split ±20交易日、事件門檻 ±0.2σ、合併窗 ±2日 | 陽性結論在微擾下消失 → 該結論是特定參數組合的產物;否證在微擾下大量翻顯著 → 否證同樣脆弱 | 兩者皆適用 |

## 產出格式

每次執行後在稽核紀錄(`稽核紀錄/{專案名}.md`)新增「方法學檢核」章節:逐項 M1~M11 標 ✅/🟡/🔴(未執行者標「未執行」),附偵測證據(file:line)與敏感度數字;最後一句話總評「陽性結論在穩健性替代方法下是否倖存」。

## 邊界聲明(誠實性條款)

1. **「最合適的統計方式」無法被證明**——本表只能做三件可證偽的事:檢驗假設是否被資料違反(M4/M6)、檢驗結論是否倖存於穩健替代(M1~M3/M11)、指出方法與研究問題的錯位(如用線性檢定否證非線性假說,只否證了線性版本)。稽核結論只能寫「未發現已知類型的不當」,禁止寫「方法正確/最優」。
2. **本表是有限清單**,涵蓋的是已知失誤模式;未知模式永遠在清單外。M9/M10 是對 unknown unknowns 的部分防禦(不需預知 bug 類型),但也非完備。
3. **涉及「學界共識做法」的判定**(如 BY 適用情境、HAC lag 慣例),執行時應 WebSearch 查證後附來源,不憑稽核者記憶宣稱。
4. 本表驗證的仍是專案內部方法選擇,不涵蓋上游資料真實性(那屬於外部真實性驗證,見記憶 feedback-audit-rigor 第5、6點)。各層驗證在稽核紀錄中須分開標示,不可互相冒充。
