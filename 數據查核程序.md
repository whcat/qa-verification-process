# 數據查核程序(模式A,可重用範本)

稽核方固定範本。本專案的稽核分兩種獨立模式,**可分開執行、互不取代**:

- **模式A(本檔)— 數據查核**:驗證「文件聲稱的數字」是否等於「程式碼實際算出的結果」。抓的是:文件造假、數字誤植、輸出與敘述不一致、hardcode 偽裝計算。
- **模式B(見 `方法學檢核表.md`)— 方法學確認**:驗證「算出這些數字的方法」本身是否恰當、有無更穩健替代。抓的是:方法錯用、家族界定錯誤、分布假設不符、選擇性揭露。

兩種模式的關係:模式A通過只代表「數字是真的算出來的」,不代表「方法是對的」;模式B通過只代表「方法合理」,不代表「文件沒抄錯數字」。完整稽核 = 兩者都跑;單獨跑任一模式時,紀錄中必須標明另一模式尚未執行。

## 執行步驟

### D0 — 開場定錨(必做)
1. 記錄被稽核專案當下的 git commit hash(`git log -1 --format=%H`);無版控則記錄關鍵檔案的修改時間戳
2. 與使用者確認查核範圍與抽查深度(全面重跑 / 抽查為主 / 僅靜態審查)
3. 用 TaskCreate 列出完整範圍清單(依研究線/模組/資料檔分項),逐項追蹤 pending → verified / found-issue

### D1 — 重現執行
1. 從原始來源資料**獨立重新執行**對應腳本(不是只讀輸出 CSV 比對格式)
2. 輸出導向暫存目錄或改名,**絕不覆蓋被稽核專案的原始輸出檔**;若不慎覆蓋,立即 diff 確認一致並還原
3. 需要即時網路的腳本(爬蟲類)不重跑,改核對現有輸出的內部邏輯一致性,並在紀錄中如實標 ❌ 無法離線驗證
4. 記錄執行環境(Python/套件版本)

### D2 — 三方比對
逐項比對:文件敘述 ↔ 輸出 CSV ↔ 本次重跑結果。三者一致 → ✅;文件與重跑不符但 CSV 與重跑一致 → 文件層問題;CSV 與重跑不符 → 資料層問題(更嚴重,需查產出歷程)。

### D3 — 委派紀律(使用 subagent 時)
1. fork/subagent 回傳後,先檢查 usage(tool_uses、duration_ms)是否與聲稱工作量相稱——工具呼叫數過低但聲稱完成大量查核,退回重做
2. 對 subagent 回報的「✅通過」清單,稽核方本人抽樣 10~20% 獨立重算複核(正反面對稱覆核,不能只複核有問題的項目)
3. **模型分配**(依子任務輸出是否機器可驗證):
   - 機械型重跑比對(輸出可 diff/重算驗證)→ 委派給指定 `model: "sonnet"` 的全新 agent(成本約旗艦模型 1/3~1/5;注意 fork 永遠繼承主對話模型無法降級,須用 fresh agent 並在 prompt 重新交代上下文);不用 Haiku(無監督委派不用最弱模型,且 200K context 對讀大量檔案偏緊)
   - 判斷型審查(方法學、家族界定等無法機器驗證的輸出)→ 主對話旗艦模型內聯執行,不委派
   - **雙模型交叉驗證只用於「高風險陽性結論的判斷型審查」**:可部分去相關稽核方自身的幻覺/誤讀,但對上游資料錯誤與方法論共同盲點無效(同家族模型盲點相關),對機器可驗證的數字比對是浪費——一個 diff 比第二個模型便宜且可靠。雙模型一致不得寫成「已驗證」,只能寫「兩個獨立執行個體判斷一致」

### D4 — 記錄
1. 問題依 CLAUDE.md 格式:【問題】【證據】【影響範圍】【建議修正】+ 🔴/🟡/🔵 分級
2. 寫入被稽核專案內的 `docs/VERIFICATION_LOG.md`(若該專案接受)**與**稽核方自己的 `稽核紀錄/{專案名}.md`(必寫,含已核實的正確數字與下次覆查提醒)
3. **不修改被稽核專案的任何原始檔案**——只記錄、只建議

## 邊界聲明

模式A驗證的是專案內部一致性(文件↔程式↔資料檔),不涵蓋:
- 方法選擇是否恰當 → 模式B
- 上游爬蟲資料是否真實/過期 → 外部真實性驗證(需 WebFetch 對照官方來源,視預算另行啟動)

三層驗證在稽核紀錄中分開標示,不可互相冒充。
